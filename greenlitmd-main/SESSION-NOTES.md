## 2026-07-09 — The de-id gate boundary is "the redacted JSON," but the seam is "the outbound prompt"
Three independent leaks share one root cause: `assertDeidentified` runs on the redacted chart JSON, then prompt assembly appends raw material after the gate (objective_measurements + BMI/ASA on both letter paths; physician supplements on denial-fix). Why it mattered: the STATUS-ROADMAP invariant says "every seam that sends PHI-bearing text off-server," and the seam is the final string handed to `callAnthropicWithRetry`. Lesson (for Phase 2 and future routes): assert the assembled prompt, not its ingredients.

## 2026-07-09 — 60/60 SOURCE LOCK PASS has a designed-in blind spot on the denial-fix tier
The eval's only denial-fix supplement is `{ conservative_treatments_attempted: "Added PT notes" }` (`scripts/eval-pipeline.ts:840`) — no dates/durations/dosages/implant terms, and a key the route's merge doesn't even recognize. Lesson: a green eval only covers the inputs it sends; when a bug class is found, extend the eval input before trusting the re-run.

## 2026-07-10 — Phase 2: scope the de-id assert to PHI-bearing content, not the whole prompt
Fixing A1 (raw `objective_measurements`/BMI/ASA trailer strings bypassing the de-id gate) by asserting the FULL assembled prompt seemed like the obvious fix — but the prompt also contains `requestDetails.providerName`/`practiceName`/`payerName` and the letter's own dateline (`today`), none of which were ever meant to be redacted (they identify the requesting clinician/payer/today's date, not the patient). Asserting the whole prompt made deid-verify's residual scanner flag the physician's own name as `unclassified_residue` and today's date as `date_full` — a live eval run crashed on the very first generate-pa call. **Fix: assert only the concatenation of chart-JSON + the newly-covered trailers, never the requestDetails/dateline boilerplate.** Lesson: "assert the full X" is a good instinct but "full" needs a real boundary — the boundary is "everything derived from patient chart data," not "everything in the outbound string." Caught by actually running the live eval, not by code review alone.

## 2026-07-10 — A live eval failure can be in the eval harness, not the product
After fixing A3 (merge supplements into extraction before verifySourceLock) and adding a realistic supplement to the eval (C4), every `regenerate-denial-fix` run still failed on the exact scenario the fix was supposed to close. Root cause: `scripts/eval-pipeline.ts`'s `runRegenerateDenialFixRoute` correctly merged supplements before calling `finalizeLetter`, but then `return`ed the STALE pre-merge `extractionJson` — and the eval's own separate `combinedSourceLockCheck` re-verifies against whatever `extracted` comes back from that function, so it was silently checking against unmerged data regardless of what `finalizeLetter` internally did. The real production route was correct throughout (it returns `updatedExtractionJson`, derived from the merged extraction). Lesson: when a fix doesn't take effect in an eval, verify the fix directly (a cheap, no-API unit-level repro of the exact functions in question) before assuming the product code is wrong — the harness itself can be the bug, especially when it duplicates logic instead of importing the real implementation.

## 2026-07-10 — cpt_code_valid correction needs a deterministic Re: line reconciliation pass
Routing a `cpt_code_valid` supplement into `effectiveRequestDetails.cptCode` (B1 fix) correctly updates the DOCX header/score/Re:-line-generation-context, but the model is explicitly told NOT to touch the Re: line's CPT text (to avoid unpredictable rewrites), and by regenerate time `sanitizeLetterPlaceholders`' `[CPT Code]` bracket has long since resolved to the literal OLD code, which "copy verbatim" instructions then preserve. Caught by the `docx-export-verifier` agent, not by the live eval (which doesn't check DOCX-level consistency). Fixed with a deterministic post-`finalizeLetter` string replace (`\bOLDCPT\b` → `newCPT`) scoped to the returned letter text only. Lesson: a "route the correction into config, let the existing pipeline handle it" fix still needs to be checked against every place that config value is echoed as literal text already baked into unchanging content.

## 2026-07-10 — `applyKnownMap` tokens must be merged into `phiState.map`, or reidentify() can't resolve them
Found by the `phi-reviewer` agent (pre-existing since commit `582ab07`, not introduced this session, but living in code this session's fixes touched): `generateLetterFromExtraction` embeds extraction-phase PHI tokens into the letter-prompt JSON via `applyKnownMap(chartDataForRedaction, phiMap)` — a literal substring swap — but never copied those token→value pairs into `phiState.map`. If the model echoes one of those tokens verbatim (which `letter-system-prompt.ts`'s date-fidelity rule explicitly asks it to do), `reidentify()` can't resolve it back to the real value, and `sanitizeLetterPlaceholders`'s bracket-cleanup pass then silently DELETES the entire line/paragraph containing the unresolvable token — no error, no `sourceLockWarning` (a deletion isn't "ungrounded"). Fix: `Object.assign(phiState.map, phiMap)` right after `seedCountersPastKnownMap`. Lesson: two independent redaction mechanisms feeding the same downstream reidentify() step must share one map, or the "consistent" half silently isn't.
